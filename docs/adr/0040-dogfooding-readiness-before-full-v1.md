# Prioritize dogfooding readiness before full v1 polish

Spool should become useful enough to build Spool with Spool as quickly as possible. Before completing every v1 feature, we will target a Dogfooding Readiness slice: init/config, one local-worktree Task Queue, delegation or temporary task creation, single-worker `spool work --once` with Pi Launcher, Local Worktree setup, a minimal Spool Pi Extension for task lookup/workpad/status/child-task/transition updates, and enough status visibility to run real Spool development tasks.

Dogfooding may use a deliberately marked Bootstrap Task Creation shortcut before the full Delegation Session is ready, but the operator-facing concept is File-backed Task Creation. The preferred CLI shape is `spool task create --queue <key> --from-file task.md`, where `task.md` has YAML front matter for structured fields (`title`, `acceptance_criteria`, `validation_items`, optional `priority`, `state`, `tags`, `conflict_hints`, `blocking_task_identifiers`, and `review_required`) and a Markdown body for the Task Brief. The existing `spool task create --bootstrap --queue <key> --file task.md` spelling remains a compatibility path until a deliberate migration removes or aliases it. File-backed Task Creation defaults to Ready when no state is provided. Persisted Priority values remain the canonical `urgent`, `high`, `normal`, and `low` labels; File-backed Task Creation accepts only the safe alias `medium` for `normal`, emits a CLI warning when it normalizes the alias, and stores the canonical value. Operators and agents can preflight the same file format with the read-only `spool task lint --file task.md` command before mutating the Task Backend. `conflict_hints` are advisory hotspot markers only; aliases such as `anticipated_touched_files` and `touched_files` are accepted for file-backed ergonomics. Use the canonical example at `.spool/bootstrap-tasks/TEMPLATE.md` when drafting file-backed Task definitions. Dependency-aware multi-file creation extends File-backed Task Creation input handling by reading multiple file definitions, validating the same-queue dependency graph before mutation, and recording explicit blocking Task relationships. Same-batch dependencies use file-local `batch_key` values referenced by `blocking_task_keys`; existing blockers still use `blocking_task_identifiers`. It is not a separate workflow concept. Bootstrap remains the temporary dogfooding reason this path exists, not the user-facing intake model.

Dogfooding may also stop at Human Review or a completed Local Worktree with manual merge before automatic Integrating is implemented. This Manual Dogfood Merge path is an implementation sequencing compromise; the full MVP still requires Agent-Gated Integrating and squash merge.

Polish such as full review sessions, rich metrics export, pruning, and future delivery backend abstractions can follow once Spool is already helping develop itself.
