# Prioritize dogfooding readiness before full v1 polish

Tasker should become useful enough to build Tasker with Tasker as quickly as possible. Before completing every v1 feature, we will target a Dogfooding Readiness slice: init/config, one local-worktree Task Queue, delegation or temporary task creation, single-worker `tasker work --once` with Pi Launcher, Local Worktree setup, a minimal Tasker Pi Extension for task lookup/workpad/status/child-task/transition updates, and enough status visibility to run real Tasker development tasks.

Dogfooding may use a deliberately marked Bootstrap Task Creation shortcut before the full Delegation Session is ready. The v1 shortcut is `tasker task create --bootstrap --queue <key> --file task.md`, where `task.md` has YAML front matter for structured fields (`title`, `acceptance_criteria`, `validation_items`, optional `priority`, `state`, `tags`, `conflict_hints`, `blocking_task_identifiers`, and `review_required`) and a Markdown body for the Task Brief. Bootstrap defaults to Ready when no state is provided. `conflict_hints` are advisory hotspot markers only; aliases such as `anticipated_touched_files` and `touched_files` are accepted for bootstrap ergonomics. Use the canonical example at `.tasker/bootstrap-tasks/TEMPLATE.md` when drafting bootstrap files. This is an escape hatch for self-use, not the permanent intake model.

Dogfooding may also stop at Human Review or a completed Local Worktree with manual merge before automatic Integrating is implemented. This Manual Dogfood Merge path is an implementation sequencing compromise; the full MVP still requires Agent-Gated Integrating and squash merge.

Polish such as full review sessions, rich metrics export, pruning, and future delivery backend abstractions can follow once Tasker is already helping develop itself.
