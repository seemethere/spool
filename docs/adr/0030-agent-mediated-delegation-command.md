# Use `tasker delegate` for agent-mediated task intake

Tasker v1 will provide a `tasker delegate` entry point that launches a Pi-backed **Delegation Session** rather than asking humans to fill out task fields directly. The **Delegating Agent** uses a grill-with-docs-style **Delegation Interview** to clarify intent one question at a time, then creates a **Root Task** with a **Task Brief**, structured **Acceptance Criteria**, structured **Validation Items**, priority, tags, advisory **Task Conflict Hints**, optional same-queue **Blocking Tasks**, review requirement, and initial **Task State**.

The first implementation contract is documented in `docs/DELEGATION_SESSION.md`. Its CLI shape is `tasker delegate --queue <task_queue_key> "<initial human intent>"` or `--intent-file <path>` for new Root Tasks and `tasker delegate --refine <task_identifier> "<refinement intent>"` or `--intent-file <path>` for refining an existing **Backlog** Task. Refinement is intentionally limited to Backlog Tasks in the first implementation; active work continues through Worker, Review, or Operator flows.

Delegation is documentation-aware but does not edit repository docs by default. It may read context and ADR files to sharpen language, then turn needed doc changes into Acceptance Criteria, Validation Items, or follow-up Tasks so source changes still happen through the normal worktree/review/integration loop. Structured Tasker fields remain authoritative for gates and scheduling; the Task Brief and **Workpad Note** are narrative handoff surfaces, not Markdown-only replacements for requirements.
