# Use extension-native Delegation Sessions for agent-mediated task intake

Spool v1 dogfooding will prefer extension-native **Delegation Sessions**: a human-present pi session loads the **Spool Pi Extension**, the **Delegating Agent** uses a grill-with-docs-style **Delegation Interview** to clarify intent one question at a time, then calls `spool_create_delegated_root_task` to create a **Root Task** with a **Task Brief**, structured **Acceptance Criteria**, structured **Validation Items**, priority, tags, advisory **Task Conflict Hints**, optional same-queue **Blocking Tasks**, review requirement, and initial **Task State**.

The first implementation contract is documented in `docs/DELEGATION_SESSION.md`. The CLI shapes `spool delegate --queue <task_queue_key> "<initial human intent>"`, `--intent-file <path>`, and `spool delegate --refine <task_identifier> ...` remain wrapper/fallback paths around the same deterministic Spool API helpers, not the preferred dogfood conversation host. Refinement is intentionally limited to Backlog Tasks in the first implementation; active work continues through Worker, Review, or Operator flows.

Delegation is documentation-aware but does not edit repository docs by default. It may read context and ADR files to sharpen language, then turn needed doc changes into Acceptance Criteria, Validation Items, or follow-up Tasks so source changes still happen through the normal worktree/review/integration loop. Structured Spool fields remain authoritative for gates and scheduling; the Task Brief and **Workpad Note** are narrative handoff surfaces, not Markdown-only replacements for requirements.
