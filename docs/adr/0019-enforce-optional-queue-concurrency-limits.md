# Enforce optional queue concurrency limits in Spool

Spool will support an optional Queue Concurrency Limit and enforce it during claim operations, even though Symphony may also apply a global worker limit. Keeping the queue limit in Spool protects shared resources when multiple Symphony instances poll the same queue, while leaving the field optional avoids adding concurrency policy to simple deployments.

The limit counts active Agent Runs, not only Tasks in the coding portion of the lifecycle. During dogfooding this means an active Agent Run for an Integrating Task consumes the same queue capacity as an active Agent Run for Ready, In Progress, or Rework work. This intentionally keeps Local Worktree Delivery and recovery attempts inside the same local resource budget: delivery can run tests, hold repository locks, and mutate the Managed Source Repository through the Delivery Adapter.

Spool does not reserve separate Integrating-only capacity in v1 dogfooding. If Integrating work saturates a queue and blocks new Ready work, the operator should inspect the active Agent Runs, wait for completion or Claim Lease expiry, fail stuck runs with an explicit reason, or raise/clear the Queue Concurrency Limit only when local CPU, disk, and repository-lock constraints permit. Operator status output should make this tradeoff visible, including when active Integrating runs are saturating the limit.
